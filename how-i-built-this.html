How I built this

Comment j’ai construit ce site et ses outils IA

J’ai construit ce portfolio et les outils PM avec une approche simple : des interfaces statiques rapides à charger, une couche proxy sécurisée pour les appels IA, puis une orchestration plus intelligente pour guider les réponses.

Architecture globale

1) Frontend statique

Chaque outil est une page HTML/JS autonome, déployée sur GitHub Pages. Aucun secret n’est exposé dans le navigateur.

2) Proxy Cloudflare

Les appels passent par un Worker (/orchestrate) qui centralise la sécurité, le routage et les fallbacks.

3) Orchestrateur LangGraph

Un service FastAPI + LangGraph produit un plan structuré puis une réponse concise via un flux planner → synthesis.

02 Étapes de création
1
Idée & cadrage
Partir d'un vrai problème PM
Chaque outil répond à une friction réelle que j'ai vécue en tant que PM : rédiger des OKRs flous, synthétiser des entretiens à la main, préparer une roadmap pour des dirigeants. Le site est né de cette liste de frustrations, pas d'une envie de faire un portfolio classique.
2
Design
Système de design en CSS variables
Dark theme pensé dès le départ avec un système de tokens CSS (couleurs, surfaces, borders) cohérent sur toutes les pages. La palette bleue-violette-cyan crée une identité visuelle forte sans recourir à un framework UI. Typographie Syne pour les titres, DM Sans pour le corps.
3
Développement
Pages HTML autonomes, JS vanilla
Chaque outil est une page HTML indépendante avec son propre CSS et JS. Pas de routing, pas de state management, pas de composants partagés. Simple, lisible, et débuggable sans toolchain. La gestion bilingue FR/EN est faite avec un objet de traductions et des attributs data-i18n.
4
Intégration LLM
Proxy Cloudflare pour sécuriser la clé
Les pages font des requêtes vers un Cloudflare Worker qui proxifie vers l'API Groq. La clé API ne transit jamais côté client. Le Worker valide l'origine, gère les headers CORS et retourne la réponse. Déployé en quelques minutes avec Wrangler CLI.
5
Déploiement
GitHub Pages, push = deploy
Le repo GitHub est directement connecté à GitHub Pages. Chaque git push sur main déclenche un déploiement automatique en moins de 60 secondes. Zéro configuration de CI/CD complexe, zéro coût d'hébergement.

04 Décisions d'architecture
Pourquoi HTML vanilla plutôt que React ou Next.js ?
Zéro dépendance, zéro toolchain, zéro dette
Un site statique de portfolio et d'outils simples n'a pas besoin d'un framework SPA. HTML vanilla se déploie directement, se lit facilement et ne génère aucune dette technique. Moins de complexité = plus de vélocité.
Pourquoi un proxy Cloudflare Workers plutôt qu'exposer la clé API ?
Sécurité non négociable, plan gratuit généreux
Exposer une clé API dans du code client-side, c'est l'offrir à n'importe qui. Le Worker intercepte les requêtes, injecte la clé côté serveur et valide l'origine. Plan gratuit Cloudflare : 100 000 requêtes/jour — largement suffisant.
Pourquoi Groq plutôt qu'OpenAI ou Anthropic directement ?
Vitesse d'inférence et plan gratuit
Groq tourne les modèles Llama 3 sur du matériel LPU optimisé pour l'inférence. Résultat : des réponses en 1-2 secondes là où d'autres providers prennent 5-10s. Et le plan gratuit couvre amplement un usage personnel.
Pourquoi GitHub Pages plutôt qu'un VPS ou Vercel ?
Gratuit, simple, CI/CD natif
Pour des fichiers statiques, GitHub Pages est imbattable : gratuit, CDN mondial, HTTPS automatique, et le déploiement se déclenche simplement avec un git push. Pas besoin d'une infrastructure plus complexe pour ce cas d'usage.

Ce qu'on a ajouté récemment
Une route Worker POST /orchestrate qui tente d'abord l'orchestrateur externe quand il est disponible.
Un fallback natif vers Groq côté Worker si l'orchestrateur n'est pas joignable.
Un parsing plus robuste des sorties planner (y compris JSON entouré de blocs ```json).
Des tests automatiques sur l'orchestrateur et le proxy pour sécuriser les cas de routing/fallback.
Pourquoi LangGraph ?
Plan PM structuré + réponse claire, site rapide et keyless
L'objectif est d'avoir une expérience fiable et actionnable : plan PM structuré + réponse claire, tout en gardant un site rapide, keyless côté utilisateur, et simple à maintenir.
