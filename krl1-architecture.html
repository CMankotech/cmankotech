Comment KRL1
décide quoi répondre

KRL1 n'est pas un simple chatbot connecté à un LLM. Chaque réponse passe par 4 couches de décision — du contexte de page à l'orchestration LLM. Voici la logique produit derrière chaque choix.

Les 4 couches de décision
01
Détection du contexte de page
KRL1 lit l'URL dès l'ouverture et adapte son message d'accueil à l'outil PM en cours d'utilisation. 6 messages différents selon la page — OKR Builder, Discovery Assistant, Backlog Prioritizer, etc.

Contextualisation
02
KB fast-path — matching local
Avant tout appel API, KRL1 cherche un pattern dans sa base de connaissances locale (~10 catégories). Si un mot-clé matche, la réponse est servie instantanément depuis le client — 0 latence réseau, 0 coût LLM.
keyword matching FR + EN rules chip suggestions
Performance & coût
03
Appel LLM via Groq + system prompt
Si aucun pattern KB ne matche, la question est envoyée à l'API Groq (llama-3.3-70b sur LPU) avec le system prompt complet + l'historique de conversation. Un Cloudflare Worker proxy sécurise la clé API côté serveur.
Groq API llama-3.3-70b Cloudflare Worker conversation history
Intelligence générative
04
PM Journey — injection post-résultat
Un MutationObserver surveille l'apparition du div #results sur chaque outil PM. Dès qu'un résultat est généré, KRL1 injecte automatiquement un CTA "prochaine étape" dans le flux — sans intervention manuelle de l'utilisateur.

Le system prompt annoté
Tu es KRL1, l'assistant portfolio de Carlin Mankoto, AI Product Manager. Réponds de façon concise et professionnelle en français par défaut (en anglais si on te parle en anglais).
Persona & contrainte de langue Le LLM doit savoir qui il est avant tout. La contrainte de langue évite des réponses mixtes FR/EN sur les questions ambiguës. Décision : nommer l'assistant "KRL1" renforce l'identité de marque plutôt que "un assistant IA générique".
Ne réponds qu'aux questions liées à Carlin ou à ses outils. Si la question est hors sujet, redirige poliment.
Scope guard Sans cette règle, le LLM répond à tout — recettes de cuisine, code Python, etc. Le scope guard maintient KRL1 focalisé sur son objectif : conversion recruteur. Décision : "redirige poliment" plutôt que "refuse" — moins agressif pour l'UX.
PROFIL : - 3 ans d'expérience PM/PO : OAIO/ Infotel, AXA (AI PO, SecureGPT RAG), Airbus D&S (JO Paris 2024), Casino - PSPO I, SAFe POPM + SSM, Product School, Elements of AI - Master 2 Innovation Digitale (Paris-Saclay) - LLM, RAG, Prompt Engineering, Agentic AI, CrewAI, n8n, Make, Vibe Coding (Claude Code, Lovable)
Contexte en raw text Le CV complet est injecté en clair dans le contexte. Le LLM n'a jamais à inférer ou deviner — 0 risque d'hallucination sur les faits biographiques. Décision : raw text plutôt que JSON structuré — les LLMs consomment mieux le langage naturel pour ce type d'information factuelle.
INSTRUCTION CRITIQUE : Quand l'utilisateur mentionne un sujet PM (backlog, OKR, discovery, roadmap, user story, epic), inclus TOUJOURS un lien HTML cliquable vers l'outil PM correspondant. Réponse actionnable et concise (max 200 mots). N'utilise jamais de tirets longs (—). N'utilise jamais l'élision 'l'' devant un nom d'outil.
Funnel & règles de rendu Chaque réponse PM devient un point d'entrée vers les 6 outils — c'est du product-led growth par prompt. La limite de 200 mots évite les réponses essay. Décision : les règles typographiques (tirets, élision) sont des guardrails qualité découverts après test — pas d'a priori, ajoutés itérativement.

Décisions produit & trade-offs
Décision 01
KB local avant LLM — pourquoi ne pas tout envoyer à l'API ?
95% des questions sont récurrentes (qui es-tu, contact, outils). Un appel Groq coûte ~0.5s de latence et un quota API. Le KB répond en <10ms depuis le navigateur. Le LLM est réservé aux questions réellement ouvertes.
Décision 02
Cloudflare Worker plutôt qu'un backend Node/FastAPI
Un Worker serverless résout le seul vrai problème : ne pas exposer la clé API côté client. Zéro infra à maintenir, 100 000 req/jour sur le plan gratuit, déployé en 2 min avec Wrangler. Un backend complet aurait été over-engineered.
Décision 03
MutationObserver pour le PM Journey — pourquoi pas un event ?
Les outils PM sont des pages indépendantes avec des structures DOM différentes. Un event custom aurait nécessité de modifier 6 fichiers. Le MutationObserver observe #results depuis krl1-widget.js sans toucher aux pages — couplage zéro.
Décision 04
Profil en raw text dans le prompt — pas de RAG ?
Le profil de Carlin tient en ~300 tokens. Un système RAG aurait ajouté une couche d'embedding + vector DB pour récupérer les mêmes informations. Règle YAGNI : la solution la plus simple qui fonctionne est la bonne.
Décision 05
Groq + llama plutôt qu'OpenAI GPT-4 ?
Pour un assistant portfolio, la latence perçue est plus importante que la profondeur de raisonnement. Groq sur LPU donne 1-2s de réponse vs 4-6s sur GPT-4. llama-3.3-70b couvre largement les besoins pour des questions PM + profil.
Décision 06
Auto-ouverture après 6 secondes — pourquoi ce délai ?
Le visiteur a besoin de temps pour lire le hero avant qu'une popup ne s'ouvre — trop tôt crée de la friction. 6s correspond au temps moyen de lecture du titre + subtitle mesuré sur sessions analytics. L'engagement monte sans nuire à l'expérience.
